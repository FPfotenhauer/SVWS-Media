# ADR 001: Grundlegende Architektur von SVWS-Media

- Status: Akzeptiert
- Datum: 2026-03-22
- Entscheider: Projektteam SVWS-Media
- Tags: Architektur, PHP, SQLite, Sicherheit, Betrieb

## Kontext

SVWS-Media ist eine schlanke, serverseitig gerenderte Webanwendung zur Medienverwaltung im schulischen Umfeld.
Die Anwendung soll lokal und in kleinen bis mittleren Schul-Deployments stabil, wartbar und ohne komplexe Infrastruktur betreibbar sein.

Ziele:

- geringer Betriebsaufwand
- einfache Installation und Sicherung
- nachvollziehbare Fachlogik fuer Medien, Ausleihe und Rueckgabe
- Integration mit SVWS-Daten (Sync)
- solide Sicherheitsbasis fuer Session-basierte Webanwendungen

## Entscheidung

Die Grundarchitektur folgt einem modularen Monolithen mit klaren Schichten:

- Praesentationsschicht: PHP-Seiten in public/ und Templates in templates/
- Anwendungslogik: Module und Services in src/modules/
- Infrastruktur: Konfiguration, Datenbankzugriff und Migrationen in src/config/
- Authentifizierung/Autorisierung: zentrale Funktionen in src/auth/
- Persistenz: SQLite-Datei in data/

Systemkontext (vereinfacht):

Browser -> PHP Web App -> SQLite
								 |
								 +-> SVWS API (synchroner Import ueber HTTP)

## Architekturprinzipien

1. Server-side first
	 HTML wird serverseitig erzeugt; JavaScript bleibt optional und leichtgewichtig.
2. Modulare Fachlogik
	 Fachliche Regeln liegen in Services/Modulen, nicht in Templates.
3. Single Source of Truth in SQLite
	 Relevante Betriebsdaten werden in einer lokalen SQLite-Datenbank gehalten.
4. Security by default
	 Sessions, Rollenpruefung, CSRF-Schutz und Passwortregeln sind zentrale Basiskomponenten.
5. Pragmatismus vor Over-Engineering
	 Keine verteilte Architektur, solange Last- und Teamgroesse dies nicht erfordern.

## Bausteinsicht (Level 1)

- public/
	HTTP-Einstiegspunkte (Routing ueber konkrete PHP-Seiten).
- templates/
	Wiederverwendbare Layout- und UI-Elemente.
- src/auth/
	Login-Flow, Session-Handling, Rollenpruefung, Passwortwechsel.
- src/modules/media/
	Medienkatalog, Titel/Exemplar-Logik.
- src/modules/lending/
	Ausleih- und Rueckgabeprozesse, inkl. Gruppen-Ausleihe.
- src/modules/sync/
	Anbindung und Import von SVWS-Daten.
- src/config/
	Laufzeitkonfiguration, DB-Initialisierung, Migrationen.
- data/
	Laufzeitdatenbank und Schema-Artefakte.

## Laufzeitsicht (wichtigste Flows)

1. Login
	 Browser sendet Credentials -> Auth-Modul prueft Benutzer und Passwort -> Session wird gesetzt -> Redirect ins Dashboard.
2. Ausleihe
	 Nutzer scannt/barcodiert Medium und Ausleiher -> Lending-Service validiert Regeln -> DB-Write in lending.
3. SVWS-Sync
	 Admin startet Sync -> Sync-Service ruft SVWS-Endpunkt auf -> Daten werden transformiert und in lokale Tabellen geschrieben.

## Persistenz und Datenhaltung

- Primäre Persistenz ist SQLite.
- Schemaentwicklung erfolgt inkrementell ueber Migrationslogik in src/config/database.php.
- Die Laufzeitdatenbank data/database.sqlite ist eine Betriebsdatei und wird nicht versioniert.
- Backups erfolgen dateibasiert (SQLite-Datei-Sicherung).

## Sicherheitsgrundlagen

- Session-basierte Authentifizierung.
- Rollenmodell mit mindestens admin/viewer.
- CSRF-Schutz fuer schreibende Aktionen (POST).
- Passwort-Policy und erzwungener Passwortwechsel fuer Default-Zustaende.
- Login-Throttling zum Schutz vor Brute-Force.

## Betriebsmodell

- Primärbetrieb auf einem einzelnen Host (PHP + Dateisystem + SQLite).
- Optional Betrieb hinter Reverse Proxy.
- Konfiguration ueber Umgebungsvariablen fuer externe SVWS-Anbindung.

## Konsequenzen

Positive Konsequenzen:

- sehr geringe Komplexitaet und schneller Start
- einfache Fehleranalyse durch klare Datei- und Modulstruktur
- leichtes Backup/Restore-Konzept

Negative Konsequenzen:

- begrenzte horizontale Skalierbarkeit durch SQLite und lokale Datei-Persistenz
- engere Kopplung an synchrones Request/Response-Muster
- bei starken Lastspitzen frueher Optimierungsbedarf

## Abgrenzung und offene Punkte

- Diese ADR beschreibt die Ziel-Grundarchitektur, nicht jedes Implementierungsdetail.
- Detaillierte Entscheidungen (z. B. Sync-Fehlerstrategie, Rollenfeingranularitaet, Reporting-Format) werden in eigenen ADRs dokumentiert.
